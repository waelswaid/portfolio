add full OpenTelemetry observability stack (traces, metrics, structured logs)

all telemetry flows through an OTel Collector to dedicated backends:
  services --> OTLP gRPC --> OTel Collector --> Tempo (traces)
                                            --> Prometheus (metrics)
                                            --> Loki (logs)
                                            --> Grafana (dashboards, localhost:3000)

=== new files ===

observability/otel-collector-config.yaml
  - receives OTLP on gRPC :4317 and HTTP :4318
  - batch processor (5s timeout, 1024 batch size)
  - exports traces to Tempo, metrics to Prometheus scrape endpoint (:8889), logs to Loki
  - health_check extension on :13133

observability/prometheus.yml
  - scrapes otel-collector:8889 every 15s

observability/tempo.yaml
  - local trace storage at /var/tempo/traces with WAL

observability/loki.yaml
  - local log storage, tsdb store, v13 schema
  - structured metadata enabled (needed for OTel log attributes)

observability/grafana/provisioning/datasources/datasources.yaml
  - auto-provisions Prometheus, Tempo, Loki datasources with explicit UIDs
  - Tempo linked to Prometheus (trace-to-metrics) and Loki (trace-to-logs)
  - Loki derives traceID field linking back to Tempo (log-to-trace correlation)
  - anonymous admin access enabled for dev convenience

services/shared/observability.py
  - init_observability(service_name, engine=None): single entry point for all OTel setup
  - creates TracerProvider, MeterProvider, LoggerProvider with OTLP gRPC exporters
  - JSON log formatter on stdout with trace_id/span_id injected into every log line
  - logs also sent to collector via OTLP LoggingHandler (dual output: stdout + collector)
  - auto-instruments FastAPI (always), SQLAlchemy (if engine provided),
    botocore and redis (if installed, silently skipped otherwise)
  - shutdown_observability(): flushes and shuts down all three providers

services/shared/metrics.py
  - custom metric definitions on a shared meter ("portfolio.custom")
  - chat: ws_active_connections (UpDownCounter), ws_messages_total,
    kafka_messages_produced, kafka_messages_consumed, kafka_consumer_errors
  - upload: upload_requests_total, upload_bytes_total, rate_limit_hits

services/shared/tracing.py
  - KafkaHeaderCarrier: adapts Kafka headers (list of (key, bytes) tuples)
    to the OTel TextMap propagator interface (get/set/keys)
  - inject_trace_context(): serializes current span context into Kafka headers
    for the producer side
  - extract_trace_context(): deserializes trace context from Kafka headers
    on the consumer side, returns an OTel Context
  - ws_span(): async context manager for manual WebSocket span creation
    (FastAPI auto-instrumentation does not cover WebSocket endpoints)
    marks span as ERROR and records exception on failure, then re-raises

services/shared/health.py
  - check_postgres(engine): runs SELECT 1 via async engine
  - check_redis(redis_url): creates temporary async redis connection, PINGs
  - check_kafka(bootstrap_servers): connects a temporary AIOKafkaProducer (3s timeout)
  - build_health_response(checks): returns dict with "ok" or "degraded" status
    always returns 200 (not 503) to avoid Docker healthcheck restart loops
    when a dependency is temporarily unreachable

services/chat/kafka/producer.py
  - creates "kafka.produce chat-messages" span (SpanKind.PRODUCER) around send
  - sets messaging.system and messaging.destination attributes
  - injects trace context into Kafka message headers via inject_trace_context()
  - increments kafka_messages_produced after send

services/chat/kafka/consumer.py
  - extracts trace context from Kafka message headers via extract_trace_context()
  - attaches extracted context so consumer span becomes child of producer span
  - creates "kafka.consume chat-messages" span (SpanKind.CONSUMER)
  - context detached in finally block
  - _handle_message now returns bool (True=success, False=dropped)
  - span marked ERROR when message is dropped after max retries
  - kafka_messages_consumed incremented for all consumed messages
  - kafka_consumer_errors incremented only on max retry exhaustion

=== end-to-end trace path for a chat message ===

ws.handle "message" (WebSocket span, manual)
  --> kafka.produce "chat-messages" (producer span, manual, injects headers)
    --> kafka.consume "chat-messages" (consumer span, manual, extracts headers)
      --> INSERT INTO messages (SQLAlchemy span, auto-instrumented)
